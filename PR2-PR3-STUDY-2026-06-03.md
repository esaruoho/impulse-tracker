# What PR #2 and PR #3 did — study (2026-06-03)

Both PRs are about ONE feature: **F12 → "Initialise Instruments? = YES"**
(`F_SetControlInstrument` in IT_F.ASM), the Samples→Instruments conversion.
This IS the "convert to instruments when switching from samples to instruments,
retaining the content" behaviour you want scalpeled out. "Retaining the content"
= retaining drawn envelopes.

Both PRs carried **"DO NOT MERGE"** in their titles/bodies. **Both were merged anyway.**

---

## Background: the crash class

`F_SetControlInstrument` (F12, answer YES to "Initialise Instruments?") historically
did, upstream IT2.15:
1. `Music_ClearAllInstruments` (wipe all 100 instrument slots), then
2. walk slots 0..99: copy sample name → instrument name, fill the 120-note keymap.

A fork feature (commits `d8ec842` etc.) tried to **preserve envelopes** the user drew
in Sample mode by NOT clearing slots whose envelope bytes differed from the template.
Problem: instrument slots routinely hold **uninitialised garbage** (from Sample-mode
loads and **Shift-Enter bulk-load**). That garbage "differed from the template", so it
was preserved and then fed to the envelope renderer → unbounded node loop → **EMM386
exception #12 wild write** = the hard crash. (Only reproduces on real DOS hardware with
EMM386; DOSBox-X can't trigger it.)

## PR #2 — `b5a0c66` "Remove envelope preservation; gate Shift-F4" (merged 2026-06-02)

**Did:** REVERTED the envelope-preservation feature. Net **+37 / −166 lines**.
- `IT_F.ASM`: `F_SetControlInstrument` restored to upstream unconditional
  `Music_ClearAllInstruments` + clean 0..99 remap. Dropped the per-slot
  `Music_InstrumentHasEnvelopes` / `Music_ClearInstrument` logic.
- `IT_MUSIC.ASM`: deleted `Music_InstrumentHasEnvelopes` (left a tombstone).
- `IT_G.ASM`: gated `Glbl_Shift_F4` to Instrument mode (bails with an info-line hint
  in Sample mode). *(This gate was later removed again in this session's `8c32fd2`
  when Shift-F4 became the 3-state cycle.)*
- KEPT: the `I_MapEnvelope` MaxNode≤25 clamp (defensive insurance).

**Introduced:** nothing new behaviourally — it took the crash-class feature OUT and
returned F12 to safe upstream behaviour. Said "DO NOT MERGE yet — opened for an audit
pass," but was merged ~24 min later.

---

## PR #3 — `c2094e6` → `a44a607` → `9a1142c` "retain drawn envelopes" (merged 2026-06-03, TODAY)

**Did:** RE-INTRODUCED the very feature PR #2 removed — but rebuilt on a better signal.
Net **IT_F.ASM +89/−9, IT_MUSIC.ASM +52/−6**, plus a 194-line notes file.

The new design (live on main now, `IT_F.ASM:4920`):
- New helper **`Music_InstrumentIsReal`** — ZF=1 iff the header starts with the 4-byte
  `IMPI` magic (every initialised instrument has it; garbage never does).
- `F_SetControlInstrument` YES-loop, per slot 0..98:
  - `Music_InstrumentIsReal`? **No (garbage)** → `Music_ClearInstrument` to a valid
    blank template first (so the renderer never sees a non-IMPI header).
  - Then, if a matching sample exists → write instrument name + 120-note keymap.
  - The **envelope section (130h+) is NEVER written** → drawn envelopes always survive.

**Introduced:**
- The IMPI-gated clear (theoretically closes the crash at the signal: garbage can't
  reach the envelope renderer).
- Envelope retention across the Sample→Instrument flip.
- **A trade-off:** a hand-crafted instrument's custom name/keymap IS overwritten by the
  auto-mapping on Initialise=YES (answer **No** to keep everything).

**The governance problem:** PR #3's body says **"Still DO NOT MERGE without real-hardware
verification"** — the EMM386 #12 crash cannot be reproduced under DOSBox-X, so the fix is
**only build-verified, never confirmed on the hardware where the bug actually bites.**
It was merged today anyway. So the crash-class feature PR #2 deliberately removed is back
on main, unverified on the platform that crashes.

---

## Why this matches your worry

You said the conversion makes sample loading brittle, "especially shift-enter loading."
That is exactly the documented failure mode: **Shift-Enter bulk-load leaves garbage in
instrument slots**, and the Samples→Instruments conversion is the thing that historically
turned that garbage into an EMM386 #12 crash. PR #3 claims to neutralise it via the IMPI
check — but that claim is **unverified on real hardware**, and it re-couples instrument
state to the loaders again.

---

## The two honest options

1. **Scalpel out PR #3 again** — revert `F_SetControlInstrument` to PR #2's safe upstream
   clear+remap (no envelope retention, no per-slot IMPI dance). Loading stops touching
   instrument state through this path. This is the "yet again" you proposed. Lose: drawn-
   envelope retention across the flip.
2. **Keep PR #3 but verify it** — runtime-test on real DOS+EMM386: bulk-load a module via
   Shift-Enter, then F12 Initialise=YES, confirm no crash and envelopes survive. Only flip
   to "trusted" after that passes. Until then it stays a known risk.

My read: given your stated worry and that PR #3 went in unverified against its own
DO-NOT-MERGE instruction, option 1 (revert again) is the defensible default — with a
clean report card documenting why, so it isn't silently re-introduced a third time.
